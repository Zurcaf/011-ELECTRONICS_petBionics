\chapter{Preliminary Work}
\label{chapter:preliminary}

This chapter presents the work carried out during the first phase of the project,
covering hardware assembly, firmware development, and initial validation experiments.
The results described here represent the current state of the prototype and provide
the empirical basis for the planned data acquisition campaigns.

% ----------------------------------------------------------------------
\section{Hardware Prototype}
\label{section:hw_prototype}

An initial prototype of the instrumented prosthesis has been assembled and validated.
The system integrates all sensing, processing, and storage components into a compact
embedded platform designed to fit within the structural constraints of the prosthetic device.

\subsection{Component Overview}

The hardware architecture is built around the Seeed Studio XIAO ESP32-C3
microcontroller~\cite{ESP32C3}, selected for its compact form factor
(21$\times$17.5\,mm), integrated Wi-Fi~802.11b/g/n, and low operating voltage
(3.3\,V), making it suitable for battery-powered wearable applications.

Inertial sensing is provided by the MPU-9250 nine-axis IMU~\cite{MPU9250},
which integrates a triple-axis accelerometer, a triple-axis gyroscope, and the
AK8963 three-axis magnetometer in a single package.
The device communicates over SPI and provides sufficient measurement range
($\pm$8\,g accelerometer, $\pm$1000\,°/s gyroscope) for the accelerations
expected during canine locomotion.

Force measurement is performed using an LCR02 inline load
cell~\cite{LCR02} interfaced through an HX711 24-bit
analog-to-digital converter~\cite{HX711}.
The HX711 supports a maximum output data rate of approximately 80\,Hz,
which defines the acquisition rate of the entire pipeline.
A calibration factor of $k = 18570.0$ was determined empirically during
bench testing using a reference weight, establishing the relationship
between raw ADC counts and force in kilograms.

Data is stored locally on a microSD card using the SPI
interface~\cite{JoyIT_COMMSD}, enabling high-capacity storage
independent of wireless connectivity.

The system is powered by three lithium polymer (LiPo) cells of 250\,mAh
each~\cite{LiPo250mAh}, connected in parallel to yield a total nominal
capacity of 750\,mAh at 3.7\,V.
Battery voltage is monitored continuously via an analog input pin, using a
measured scaling factor of 1.73 to correct for the ADC input divider.

\subsection{System Interconnection}

The MPU-9250 and the microSD card share the same SPI bus, with independent
chip-select lines assigned to each peripheral.
The HX711 uses a dedicated two-wire serial interface (clock and data).
The LiPo cells are connected in parallel and supply power to the
microcontroller's onboard 3.3\,V low-dropout regulator.

\section{Firmware Architecture}
\label{section:fw_architecture}

The firmware was developed using the Arduino framework for ESP32 under
PlatformIO, running on a single cooperative loop at a target acquisition
rate of 80\,Hz, constrained by the HX711 output data rate.

\subsection{Acquisition Pipeline}

The main acquisition loop executes the following steps on each sample tick:
\begin{enumerate}
  \item Read raw force value from HX711 and convert to estimated weight (kg)
      using the calibration factor $k = 18570.0$;
  \item Read nine IMU axes (accelerometer, gyroscope, magnetometer) from the
      MPU-9250 via SPI burst read;
  \item Apply a first-order low-pass filter to the force signal
      ($\alpha = 0.2$, configurable at runtime);
  \item Update the orientation estimator with the new IMU sample;
  \item Evaluate the event detector against the filtered force signal;
  \item Write the complete sample to the active CSV session file on the SD card.
\end{enumerate}

Each CSV row contains the fields listed in Table~\ref{tab:csv_fields}.

\begin{table}[h]
  \centering
  \caption{CSV data fields recorded per sample at 80\,Hz.}
  \label{tab:csv_fields}
  \begin{tabular}{lll}
    \toprule
    \textbf{Field} & \textbf{Unit} & \textbf{Description} \\
    \midrule
    \texttt{sample\_us}          & $\mu$s           & Microsecond timestamp (device clock) \\
    \texttt{time\_of\_day}       & HH:MM:SS.mmm     & Local wall-clock time (NTP-synced) \\
    \texttt{batt\_v}             & V                & Battery voltage at sample time \\
    \texttt{load\_cell\_raw}     & counts           & Raw HX711 ADC value \\

\subsection{Orientation Estimation}

Orientation is estimated using a quaternion-based complementary filter that
fuses gyroscope integration with an absolute reference derived from the
accelerometer and magnetometer.
The filter blends the gyroscope-propagated quaternion with the
accel\,+\,mag reference at each sample step using a configurable
blend factor ($\alpha_{\text{gyro}} = 0.97$), providing smooth and
drift-resistant estimates.
During fast rotations (gyroscope magnitude above a threshold), the
accelerometer reference is discarded and the filter relies exclusively
on gyroscope integration, avoiding errors from centripetal acceleration.
Final Euler angles (roll, pitch, yaw) are extracted from the quaternion
for storage and display.

\subsection{Event Detection}

A threshold-based event detector monitors the filtered force signal and
triggers an event when the signal exceeds a configurable threshold,
subject to a minimum inter-event cooldown period (default: 300\,ms).
This mechanism is intended to detect ground contact events (stance phase
onset) during locomotion.
The event count is accumulated per session and reported in the system status.

\subsection{Data Management}

Session files are created on the SD card under a hierarchical directory
structure (\texttt{/inbox/YYYYMMDD/}), with filenames encoding the
date, time, and sequential run number for unambiguous identification.
When Wi-Fi is available, the system synchronises the real-time clock
via NTP before session creation, ensuring that filenames and
\texttt{time\_of\_day} fields reflect accurate wall-clock time.
Files are flushed to the SD card every 80 samples ($\approx$1\,s)
to balance write overhead against data loss risk.

\subsection{Communication Interfaces}

Two complementary interfaces allow runtime control and data retrieval:

\textbf{Web interface.}
An HTTP server (port~80) serves a browser-accessible dashboard that
displays live system status (battery voltage, estimated force, orientation
angles, sensor health, and session statistics) updated via WebSocket at
10\,Hz.
The interface provides start/stop session controls and allows CSV files
to be listed and downloaded directly from the SD card.

\textbf{Serial debug interface.}
A command-line interface over USB CDC serial (115\,200\,baud) exposes
the following commands: \texttt{start}, \texttt{stop}, \texttt{status},
\texttt{files}, \texttt{delete}, \texttt{delete-all}, \texttt{ssid},
\texttt{wifi-on}, \texttt{wifi-off}, \texttt{wifi-reset},
\texttt{reset}, \texttt{sleep}, and \texttt{wakeup}.
Wi-Fi credentials modified via the \texttt{ssid} command are persisted
to non-volatile storage (ESP32 NVS) and restored automatically on boot.
A periodic heartbeat line is emitted every 5\,s to confirm device state
without polluting the terminal.

\section{Sensor Validation}
\label{section:sensor_validation}

Initial validation experiments were conducted to confirm correct sensor
operation and assess measurement quality before proceeding to locomotion
data collection.

\subsection{System Initialisation}

The device was powered and the boot sequence was verified over the serial
interface.
All peripherals were detected and initialised successfully:
the IMU reported \texttt{WHO\_AM\_I\,=\,0x73} (MPU-9250) and the AK8963
magnetometer reported \texttt{WHO\_AM\_I\,=\,0x48};
the HX711 and SD card also initialised without errors.
Wi-Fi connected to the local access point and NTP time synchronisation
completed within 5\,s of boot.

\subsection{Load Cell Calibration}

The HX711 was calibrated by placing known reference weights on the load cell
and recording the corresponding raw ADC output.
The calibration factor $k = 18570.0$ (ADC counts per kilogram) was derived
from this procedure and applied in firmware.
The estimated weight $\hat{w}$ is computed as:
\begin{equation}
  \hat{w} = \frac{r - r_0}{k}
  \label{eq:hx711_calib}
\end{equation}
where $r$ is the raw ADC reading and $r_0$ is the tare offset measured
with no load applied.

\section{Current Limitations}
\label{section:current_limitations}

The current prototype presents several limitations that will be addressed
in subsequent development phases.

\textbf{Load cell calibration.}
The calibration factor was determined from a limited set of reference
measurements.
A formal multi-point calibration against certified weights, covering the
full expected force range, is required before quantitative comparison
across sessions or animals.

\textbf{IMU placement sensitivity.}
Orientation and force estimates are sensitive to sensor placement and
orientation within the prosthetic housing.
A mechanical fixture ensuring repeatable positioning across sessions has
not yet been implemented.

\textbf{Dataset size.}
Data collected to date is limited to bench and simulated gait experiments.
Acquisition sessions with a canine subject are required to characterise
the actual signal environment and to validate the proposed biomechanical
metrics under real operating conditions.

\textbf{Power autonomy.}
The estimated autonomy of 6--8\,h in Wi-Fi-connected mode
(100--175\,mA average) and 12--16\,h without Wi-Fi
(35--55\,mA) has not yet been validated through a full discharge cycle.
This measurement is planned as part of the next development phase.
